Capture Methods Explained

1. OTLP Native (VS Code)

VS Code Copilot has built-in OpenTelemetry support. It sends spans following the GenAI Semantic Conventions directly to the collector on port 4318. Zero overhead, zero proxy — the gold standard.

2. HTTPS Proxy (Cursor, Eclipse, CLIs)

For IDEs and tools that don't support OTLP natively, a local mitmproxy instance intercepts HTTPS traffic to AI API endpoints (Anthropic, OpenAI, GitHub Copilot). A Python addon converts each request/response pair into an OTLP span and forwards it to the collector. The proxy runs on 127.0.0.1 only — traffic never leaves your machine.

3. REST Direct (Custom Agents)

Agents with env-var access (OpenCode, Antigravity, custom scripts) post events directly to the /events/tool-call REST endpoint. Simplest integration — just curl.

Key Design Decisions

  • Conversation grouping: Uses COALESCE(trace_id, conversation_id) to unify spans that belong to the same interaction, even across different capture methods.
  • IDE detection: Pattern-matched from user-agent headers and service.name attributes — no manual tagging required.
  • Token estimation: Extracted from usage.prompt_tokens / usage.completion_tokens in API responses, or from OTLP gen_ai.usage.* attributes.
  • Privacy-first: Only metadata (model, tokens, duration, tool names) is stored by default. Full prompt capture is opt-in.

Cursor

Cursor uses direct HTTPS connections to Anthropic, OpenAI, and GitHub Copilot APIs. A local mitmproxy intercepts these calls transparently.

One-Time Setup

cd agent-meter/cursor-proxy

# Generates CA, installs system-wide, creates systemd service
./start_proxy.sh --setup

Daily Usage

# Instead of "cursor .", use:
cursor-metered .

# Operations:
cursor-metered --status   # proxy status + recent logs
cursor-metered --logs     # live interceptor output
cursor-metered --stop     # stop the proxy
Zero friction: cursor-metered auto-starts the proxy if it's not running, sets the required env vars, and launches Cursor. One command replaces your normal workflow.

How It Works

  1. cursor-metered checks if mitmproxy is already running on port 8898
  2. If not, starts mitmdump with cursor_interceptor.py in background
  3. Launches Cursor with HTTPS_PROXY=http://127.0.0.1:8898 and NODE_EXTRA_CA_CERTS
  4. The interceptor captures all requests to AI API hosts and converts them to OTLP spans

Monitored Endpoints

HostAPIModels
api.anthropic.com/v1/messagesClaude Sonnet, Opus
api.openai.com/v1/chat/completionsGPT-4o, GPT-4.1
api.githubcopilot.com/chat/completionsGPT-4o (Copilot plan)
api2.cursor.sh/v1/chat/completionsAny via Cursor proxy

Conversation Grouping

Cursor doesn't expose a stable session ID. The collector uses a multi-strategy approach:

  1. x-session-id or x-cursor-session headers (when available)
  2. Bearer token prefix (first 16 chars — stable per login)
  3. 30-minute idle window — requests within 30 min are grouped together

Copilot CLI (gh copilot)

The GitHub Copilot CLI (gh copilot suggest, gh copilot explain) makes HTTPS calls to GitHub's API. The same mitmproxy technique captures them transparently.

Setup

# Requires: mitmproxy running on :8899 (from eclipse-proxy setup)
# The same proxy serves both Eclipse and Copilot CLI

cd agent-meter/eclipse-proxy
./start_proxy.sh

Usage

# Instead of "gh copilot suggest", use the metered wrapper:
./copilot-cli-metered.sh suggest "list all pods in kubernetes"

# Or explain:
./copilot-cli-metered.sh explain "kubectl get pods -A"

How It Works

The wrapper script injects HTTPS_PROXY and SSL_CERT_FILE environment variables, then calls gh copilot with your original arguments. All HTTP traffic is routed through the interceptor.

Agnostic by design: The mitmproxy intercepts at the HTTP layer. It doesn't matter if the request comes from a GUI IDE or a CLI tool — if it speaks HTTPS to an AI API, it gets captured. This means any new CLI tool that uses standard HTTP libraries will work automatically.

Claude Code (CLI)

Anthropic's claude CLI makes direct HTTPS calls to api.anthropic.com. The mitmproxy technique captures these identically to Cursor.

Setup

# Use the cursor-proxy (same Anthropic endpoint interception)
cd agent-meter/cursor-proxy
./start_proxy.sh --setup

Usage

# Option A: Set env vars in your shell profile
export HTTPS_PROXY="http://127.0.0.1:8898"
export SSL_CERT_FILE="$HOME/.mitmproxy/mitmproxy-ca-cert.pem"
export NODE_EXTRA_CA_CERTS="$HOME/.mitmproxy/mitmproxy-ca-cert.pem"

# Then use claude normally:
claude "explain this code"

# Option B: One-liner wrapper
HTTPS_PROXY=http://127.0.0.1:8898 \
SSL_CERT_FILE=~/.mitmproxy/mitmproxy-ca-cert.pem \
claude "explain this code"

Detection

The collector detects Claude Code via user-agent patterns (claude-code, claude_code) and assigns the claude-code IDE badge in the dashboard.

Same proxy, different tool. Since Claude Code talks to the same api.anthropic.com endpoint as Cursor's Claude integration, the cursor_interceptor.py captures both seamlessly. The IDE is distinguished by user-agent analysis.

SDKs

Official SDKs wrap the OTLP endpoint with ergonomic APIs. Both auto-flush in the background and work with any AGENT_METER_ENDPOINT.

Python

pip install agent-meter
from agent_meter import AgentMeter

am = AgentMeter(api_key="am_live_...")

# Track a tool call
span = am.track("web_search", model="gpt-4o")
result = search(query)
span.finish()

# With tokens and cost
span = am.track("code_review", model="claude-sonnet-4-20250514")
span.estimated_input_tokens = 2000
span.estimated_output_tokens = 500
span.usd_cost = 0.012
span.finish()

Node.js / TypeScript

npm install @agent-meter/sdk
import { AgentMeter } from "@agent-meter/sdk";

const am = new AgentMeter({ apiKey: "am_live_..." });

const span = am.track("web_search", { model: "gpt-4o" });
const result = await search(query);
am.finish(span);

// With tokens and cost
const span2 = am.track("code_review", { model: "claude-sonnet-4-20250514" });
span2.estimatedInputTokens = 2000;
span2.estimatedOutputTokens = 500;
span2.usdCost = 0.012;
am.finish(span2);

OTLP Protocol

Receiver

The collector accepts OTLP/HTTP on port 4318 at /v1/traces. Both protobuf and JSON encodings are supported.

Supported Attributes

OTLP AttributeMapped ToDescription
gen_ai.agent.nameagentAgent/extension name
gen_ai.tool.nametool_nameTool being invoked
gen_ai.response.modelmodelLLM model used
gen_ai.usage.input_tokensinput_tokensPrompt tokens
gen_ai.usage.output_tokensoutput_tokensCompletion tokens
copilot_chat.chat_session_idconversation_idVS Code session
gen_ai.conversation.idconversation_idGeneric session ID

Conversation ID Resolution Chain

The collector resolves conversation IDs using a fallback chain (first match wins):

  1. gen_ai.conversation.id
  2. copilot_chat.chat_session_id
  3. copilot.conversation.id
  4. thread.id
  5. gen_ai.thread.id
  6. mcp.session.id
  7. session_id
  8. trace_id (OTLP trace ID as fallback)

Security

Data Privacy

  • No prompts stored by default — only metadata: model name, token counts, tool names, durations
  • Opt-in content capture — requires explicit captureContent: true in VS Code, or custom interceptor config
  • Bearer tokens never stored — only a prefix hash is used for session grouping
  • Local-first architecture — all data stays on your infrastructure

Proxy Security

  • mitmproxy listens on 127.0.0.1 only — never exposed to the network
  • Generated CA certificates are unique per installation
  • CA can be removed at any time to disable interception
  • No data is sent to third parties — all telemetry goes to your own collector

Removing the Proxy CA

# Linux
sudo rm /usr/local/share/ca-certificates/mitmproxy-*.crt
sudo update-ca-certificates

# macOS
security delete-certificate -c "mitmproxy" ~/Library/Keychains/login.keychain

# Windows
certmgr.msc → Trusted Root CAs → delete "mitmproxy"

FAQ

Does the proxy add latency?

Negligible (<1ms). mitmproxy processes requests in-memory on localhost. The interceptor addon is non-blocking — it sends telemetry to the collector asynchronously after the AI API response is forwarded to the IDE.

Can I use the proxy for CLI tools and IDEs simultaneously?

Yes. A single mitmproxy instance on port 8898 can serve Cursor, Claude Code, Codex CLI, and any other tool that respects HTTPS_PROXY. The interceptor distinguishes tools by user-agent headers.

What if my IDE updates break the interception?

The proxy technique is transport-level (HTTP). IDE updates that change UI or features don't affect it. Only a change in how the IDE makes network requests (e.g., ignoring proxy env vars) would break interception. This is rare and easily detected — you'd see zero new spans in the dashboard.

Is this compatible with corporate proxy/VPN setups?

Yes. Configure mitmproxy with an upstream proxy: mitmdump --mode upstream:http://corporate-proxy:8080 -s interceptor.py. The interception still works — it just chains through your corporate proxy.

Can I run agent-meter without PostgreSQL?

Not currently. PostgreSQL is the only supported backend. SQLite support is on the roadmap for single-developer setups.

How do I export data?

Use the REST API to pull data programmatically, or the Reports page for pre-built views. CSV export is available from the Cost page.
